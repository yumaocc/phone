1. 你最核心的技术栈是什么？
   可以答：
   “我的主栈是 React + TypeScript，日常做得最多的是 B 端复杂业务系统开发。除了 React，我也有 Vue 2/3、React Native、Taro 的实际项目
   经验。我的优势不只是页面开发，更偏向复杂表单、组件抽象、跨端协作、性能优化和业务系统重构这几类场景。”

2. 你怎么理解 React 渲染机制？
   可以答：
   “React 本质上是状态驱动视图，组件状态或父组件 props 变化后，会触发重新执行组件函数，生成新的虚拟 DOM，再做 diff 和提交更新。实际
   开发里我更关注两个点：一是哪些更新是必要的，二是哪些子组件被无效带着一起渲染了。所以我做性能优化时，不会一上来就加 memo，而是先看
   状态边界是不是合理、组件拆分是不是清晰，再决定要不要做缓存优化。”

3. React.memo、useMemo、useCallback 分别怎么用？
   可以答：
   “React.memo 主要用于避免子组件在 props 没变时重复渲染；useMemo 适合缓存高开销计算结果；useCallback 适合稳定函数引用，避免子组件因
   为函数 props 变化而重复渲染。
   不过我实际使用会比较克制，因为这些优化本身也有维护成本。我通常会先通过 React DevTools 或页面表现确认瓶颈，再做针对性优化，而不是
   全局乱加。”

4. 你做复杂表单时，状态管理怎么设计？
   可以答：
   “复杂表单我一般先拆成三层：字段值、联动规则、提交流程。字段值负责存当前输入结果，联动规则负责处理显示隐藏、校验、依赖更新，提交流
   程负责组装最终数据和提交状态。
   如果是步骤很多、跨页面共享、还要支持回退和缓存的场景，我会把表单状态抽到统一容器里，保证页面切换后状态不丢。这样做的核心不是技术
   炫技，而是避免业务规则散落在各个组件里，后期很难维护。”

5. 你说你做过性能优化，一般怎么定位性能问题？
   可以答：
   “我一般先分层定位：先判断是请求慢、计算慢，还是渲染慢。
   如果是页面卡顿，我会先看用户操作链路，再结合浏览器 Performance、React DevTools 和日志埋点去判断瓶颈位置。比如是某次输入触发了整表
   重算，还是组件树重复渲染太多，或者是大列表 DOM 太重。只有先定位到具体瓶颈，后面的优化才有意义。”

6. 你简历里写表格性能优化，为什么能提升这么多？
   可以答：
   “那个场景的核心问题不是单纯渲染，而是数据更新策略有问题。原来的实现是改一个单元格就全表遍历，导致每次输入都触发大计算。我的优化思
   路是把‘全量重算’改成‘增量更新’，只根据当前变更值去更新关联行、列和汇总值，把计算范围压缩到最小。
   另外在渲染层引入了虚拟滚动，只渲染可视区域，减少大量 DOM 节点。最终其实是‘计算层 + 渲染层’一起优化，所以提升才会比较明显。”

7. 你提到复杂度从 O(n²) 到 O(1)，怎么理解？
   可以答：
   “这里不是严格数学意义上的所有流程都变成 O(1)，而是针对单次编辑这个核心操作，把原来依赖全表扫描的更新，改成只处理当前改单元格带来
   的差值和受影响的聚合项。
   也就是说，单次编辑不再需要遍历整张表，而是直接更新关联结果，所以从用户体感上看响应速度会明显提升。面试里最好说明这是‘单次核心编辑
   链路的复杂度优化’，这样表达更严谨。”

8. 虚拟滚动的原理是什么？
   可以答：
   “虚拟滚动的核心是只渲染可视区域附近的内容，而不是一次性把所有列表或表格节点都挂到 DOM 上。滚动时根据容器位置动态替换可见项，用占
   位高度维持整体滚动条长度。
   它适合大列表、大表格场景，能显著降低 DOM 数量和渲染压力。但它也有代价，比如动态高度处理更复杂，滚动定位、固定列、编辑态同步这些细
   节都要额外处理。”

9. 你做重构时，怎么判断是重写还是渐进式改造？
   可以答：
   “我一般看三个因素：业务是否稳定、模块耦合程度、上线风险。
   如果业务还在频繁变动，我更倾向渐进式重构，优先统一代码风格、拆组件、收敛状态管理、清理明显技术债；如果某个模块已经完全不可维护，
   而且边界相对独立，才会考虑局部重写。
   我不太赞成为了代码好看就全量推翻，重构本质上还是要服务交付效率和稳定性。”

10. React Native 和 Web 开发最大的差异是什么？
    可以答：
    “我理解最大差异有三个。第一是渲染载体不同，Web 面向浏览器 DOM，RN 面向原生组件；第二是性能关注点不同，RN 更容易遇到桥接通信、列表
    渲染和端能力调用问题；第三是交互和调试链路不同，很多 Web 上习惯的调试方式在 RN 上要结合原生环境一起看。
    所以我做 RN 时会更注意组件颗粒度、长列表性能、状态同步和原生能力边界。”

11. 微前端适合解决什么问题？
    可以答：
    “微前端比较适合多团队并行开发、系统体量大、发布节奏不一致的场景。它解决的是大型前端系统的组织和拆分问题，比如独立部署、技术栈隔
    离、团队协作边界更清晰。
    但它不是银弹，也会引入运行时通信、样式隔离、公共依赖重复加载、子应用接入规范这些成本。所以我会把它理解成工程治理方案，而不是单纯
    的技术亮点。”

12. 你做过 Monorepo，实际价值是什么？
    可以答：
    “Monorepo 对我来说最大的价值是统一管理公共代码和协作规范。比如跨端项目里，公共组件、工具函数、业务包、类型定义可以集中维护，避免
    多仓库复制粘贴。
    另外它也能帮助团队统一 lint、构建、测试和发布流程。不过 Monorepo 的前提是边界要清楚，不然仓库会越来越重，所以设计 package 结构时
    还是要有约束。”

13. 你提到 LangChain / LangGraph，你是怎么理解它们的？
    可以答：
    “我理解 LangChain 更偏向把模型调用、Prompt、工具、记忆、检索这些能力串起来，方便搭建应用；LangGraph 更适合做有状态、多步骤、可控
    制流程的 Agent 工作流。
    如果只是简单问答，LangChain 就够了；但如果要做追问、补全上下文、任务分支、状态流转这类更复杂的流程，用 LangGraph 会更清晰，因为它
    更像是在描述一个可控的状态机。”

14. 面试官问你：你做的 Agent 不就是套壳吗？
    可以答：
    “我会区分‘调用模型’和‘设计系统’。单纯接一个模型接口当然不算复杂，但如果要让系统真的可用，就要处理多轮对话上下文、信息补全、任务状
    态、结果回查、Prompt 结构化、风格检索和异常兜底。
    我理解 Agent 的价值不在于名字，而在于你有没有把模型能力嵌进业务流程里，让它稳定地产生结果。这个部分才是我更关注的。”

15. 线上出了问题你一般怎么排查？
    可以答：
    “我一般先止损，再定位。先确认影响范围、复现路径和优先级，必要时做降级或回滚。然后结合监控、日志、埋点、最近发布记录去判断是前端问
    题、接口问题还是环境问题。
    如果是前端，我会继续看是不是状态异常、边界条件漏处理、缓存脏数据、版本兼容或异步竞态导致。排查结束后我会补上对应的监控或兜底，避
    免同类问题再发生。”

16. 你怎么回答‘你的优势是什么’？

    验比较完整，能兼顾 Web、App 和公共能力建设；第三，遇到性能或架构问题时，我会先分析瓶颈，再做有依据的优化，而不是停留在表面。”
